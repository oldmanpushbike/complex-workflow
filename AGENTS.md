# 共享 Agent 契约

本文件是 Codex、Claude Code、Superpowers 以及所有专业技能的共享契约。
在复杂项目中应将其复制到项目根目录。

## 使命

通过「规格优先、审查密集」的工作流构建复杂项目：

```text
OpenSpec → grill-me → find skill → implementation → dual review → verification
```

规格是唯一的事实来源。代码只有在可追溯到规格、通过约定的验证、
且在至少一次怀疑性审查中存活下来时，才算有效。

## 双模式，同一契约

本工作流是**双向的**——任一 Agent 都可以担任编排者：

- **Codex 编排：** Codex 拥有主干，Claude 挑战和审查。
- **Claude 编排：** Claude 拥有主干，Codex 挑战和审查。

两种模式下契约、门禁和制品完全相同。仅默认实现者和启动文件不同。

## Agent 角色

这些角色是**对称的**——任一 Agent 都可以填充任何角色。
编排者是启动该功能的那个人。

### 编排者（任一 Agent）

- 拥有整个工作流，保持所有制品处于最新状态。
- 在修改代码库之前先阅读代码。
- 产出 OpenSpec 提案。
- 将 grill-me 挑战路由给另一个 Agent。
- 构建任务-技能映射表。
- 在需求明确时实现限定范围的变更（或委托实现）。
- 运行测试并记录验证结果。
- 在完成时编写 ADR 和任务回顾。

### 挑战者 / 审查者（任一 Agent）

- 挑战架构、数据流、风险和隐藏假设。
- 对照 OpenSpec 制品审查实现。
- 当编排者或 Superpowers 不可用时，作为备选实现者。
- 在做出变更时编写交接笔记。

### OpenSpec（工具或手动）

- 产出目标、非目标、用户故事、数据契约、API 契约、
  验收标准、迁移说明和验证要求。

### grill-me（工具、另一个 Agent、或手动）

- 在代码编写之前攻击提案。
- 寻找缺失的权限、边缘情况、失败模式、扩展问题、
  迁移陷阱、安全风险和模糊术语。

> **grill-me 是 Agent 领地。** 它攻击技术风险：边缘情况、安全漏洞、
> 性能、数据丢失、规格缺口、契约断裂。它**不**重新审视设计决策
> （门禁 1）——那是人类的领域，在 grill-me 开始前就已确定。

### find skill（工具或手动）

- 将每个子任务路由到对应的专业技能或专家工作流。
- 除非所选技能明确包含实现，否则不自行实现。

### Superpowers（可用时）

- 实现已批准的任务卡片。
- 不重新解读产品需求。
- 将规格缺口上报回提案或 grill-me 报告。

## 编排者选择

编排者是用户要求启动功能时对应的 Agent：

- 用户要求 Codex → Codex 编排，Claude 挑战。
- 用户要求 Claude → Claude 编排，Codex 挑战。

若用户未指定，接收到请求的 Agent 自动成为编排者。

## 跨 Agent 通信

Agent 之间通过三种通道通信，按顺序尝试：

1. **MCP**——实时工具调用（首选）。
2. **CLI**——通过 Shell 调用并捕获输出。
3. **交接文件**——`workflow/handoffs/codex-to-claude.md` 和 `workflow/handoffs/claude-to-codex.md`。

若到另一个 Agent 的所有通道都失败，编排者进入**单 Agent 模式**
（见下方降级规则）。门禁保持不变；审查标记为 `single-agent`。

## 硬门禁

### 门禁 1：规格就绪

- 提案包含目标、非目标、用户、核心流程、数据/API 契约、
  验收标准，以及明确的不在范围内事项。
- **全新功能（非对已有功能的增量修改）：在 grill-me 开始前，
  人类已审查并批准提案中的关键设计决策。** 设计决策是人类
  必须拥有的选择（规模、视觉风格、技术栈、功能范围、用户角色）——
  它们不是 grill-me 流程可以裁定的风险。以简短选项列表呈现，
  而非开放式问题。

### 门禁 2：风险就绪

- grill-me 报告已被答复，或每个未解决的风险都标记为已接受并注明负责人和日期。
- **grill-me 是 Agent 领地。** 它攻击技术风险：边缘情况、安全漏洞、
  性能、数据丢失、规格缺口、契约断裂。它**不**重新审视设计决策
  （门禁 1）——那是人类的领域，在 grill-me 开始前就已确定。

### 门禁 3：任务就绪

- 每个任务都有技能路由、可能涉及的文件、测试计划和回滚说明。

### 门禁 4：实现就绪

- 仅实现标记为已批准的任务。
- 不要将无关的重构混入实现任务中。

### 门禁 5：审查就绪

- 当两个 Agent 都可用时，两份审查均已完成。
- 若一方不可用，可用方完成审查并标记为 `single-agent`。

### 门禁 6：已验证

- 测试、手动检查或有理由的无测试说明已记录。
- 残余风险明确列出。

### 门禁 7：知识已捕获

- 具有未来影响的架构决策已记录在 ADR 中。
- 显著的失败、工具问题或有用的模式已记录在任务回顾中。

## 人类交互门禁与学习型 Checkpoint

以下列出了 Agent **必须**停下来询问人类的场景。
每个场景都应遵循**学习型 Checkpoint 六段结构**，
将每次"拦住问"转化为一次教与学的机会。

### 五个基本触发条件

| 状态 | 触发条件 | 为什么只有人类能决定 |
|---|---|---|
| S0 | 功能描述模糊——存在多种合理解读 | 尚无提案存在。grill-me 和 Superpowers 在后续触发且假定有连贯的规格。只有人类能定义要构建什么。 |
| S4 | 一个任务需要两个 Agent 都不具备的领域专业知识 | 能力缺口。继续会产出垃圾。人类必须提供知识、重写任务或手动处理。 |
| S4 | 一个任务涉及生产数据、计费、认证或部署凭证 | 权限门禁。任何 Agent，无论多能干，都不应在无明确人类批准的情况下越界。 |
| S5 | 一项任务耗时超过任务映射表中估算的 2 倍 | 标志根本性误判。人类必须重新排优先级：终止、缩减范围或接受成本。 |
| S7 | 验收标准验证失败 | 只有利益相关者可以豁免失败的验收标准。Agent 无权说"够好了"。 |

### 规则

1. **立即停止。** 在人类回答之前不要进入下一个门禁。
2. **呈现选项，而非开放式问题。** 至少两条具体路径，附带权衡说明。允许有默认推荐，但备选方案必须是真实的。
3. **记录到决策日志。** 每次问答都记入对应制品的 Decision Log 表格。
4. **48 小时超时。** 如果人类未回应且问题阻碍进展，遵循「人类不可用」协议。
5. **单 Agent 模式附加规则：** 在 Solo 模式下，grill-me 发现的所有 P2 风险也**必须**向人类确认（不仅是 P0/P1），因为没有交叉检验。

## 决策规则

- 最新已接受的规格比之前的聊天上下文优先。
- 最靠近项目的指令文件比通用工作流指南优先。
- 如果实现过程中发现需求缺口，先停止编码并更新规格。
- 如果 Agent 提出大幅度重写，需要单独的 OpenSpec 提案。
- 优先选择小的垂直切片而非大的隐藏批次。
- 优先选择已签入的制品而非临时的聊天记忆。
- 并行分支仅允许在任务映射表获批准后进行。
- 分支可以独立推进实现，但必须在审查和验证时合并回来。

## 审查指南

审查发现必须以文件、规格条款或显式行为为依据。
优先列出严重问题：

- **P0：** 数据丢失、安全漏洞、阻断构建的回归、不可能完成的迁移。
- **P1：** 用户可见的错误行为、契约断裂、关键测试缺失。
- **P2：** 在大规模部署前应修复的可维护性或边缘情况问题。
- **P3：** 打磨、命名或可选的改进。

任何审查都不应批准无法追溯到已批准任务的代码。

## 降级规则

### 工具或技能失败

若工具、Agent、MCP 服务器、插件、技能或网络调用失败，
遵循 `workflow/fallback-matrix.md`。以降低的能力继续，
除非缺失的组件对安全、凭证、生产访问或破坏性操作是必需的。

### 另一 Agent 不可达

若编排者无法联系到另一个 Agent（MCP 不可用、CLI 不可用、无交接回应）：

1. 不要停止工作流。
2. 使用 `workflow/templates/02-grill-me-report.md` 手动执行 grill-me。
3. 从已批准的任务映射表实现。
4. 执行**换帽自审**：
   - **第一轮（实现者）：** 编写代码。
   - **第二轮（怀疑者）：** 阅读 diff，尝试破坏它。寻找每一个假设。
   - **第三轮（验证者）：** 运行测试并记录结果。
5. 将每个审查制品标记为 `single-agent`。
6. 写一份交接记录供另一个 Agent 后续验证。
7. 在验证日志中记录降级情况。
8. 继续通过所有剩余门禁。

工作流绝不能仅仅因为一个 Agent 不可达而停止。

### 人类不可用

若触发逃生舱时，另一个 Agent 和人类均不可达：

1. 记录阻塞点和逃生舱触发条件，附时间戳。
2. 设置超时（默认 24 小时）。文档化所有可逆选项及其风险。
3. 若超时到期且无人类回应：
   - 编写 ADR 记录单方面决策及其理由。
   - 执行最小的可逆步骤。
   - 标记功能为 `needs-human-review`。
   - 创建后续任务，明确请求人类审查该逃生决策。
4. 若项目有升级联系人，在采取单方面行动前先升级。

### 逃生舱

若连单 Agent 模式也陷入困境：

1. 记录阻塞点。
2. 请求或记录人类决策。
3. 若决策影响架构或流程，创建 ADR。
4. 仅以最小的可逆步骤继续。

## 循环与反馈

本工作流支持「暂停→恢复」循环和操作后反馈，确保中断后能无缝接续。

- **反馈循环：** 完成每个门禁后，编排者应将结果写回制品文件，形成闭环。
  详细机制见 `workflow/feedback-loop.md`。
- **恢复协议：** 当会话中断或功能被搁置后重新启动时，编排者（无论是哪个 Agent）
  必须首先读取 `workflow/resume-protocol.md`，根据功能文件夹中已存在的制品判定当前
  处于哪个门禁，然后从该门禁继续——不得重新执行已完成的门禁。

### 反馈循环触发点

| 门禁 | 反馈动作 |
|------|----------|
| 门禁 1（规格就绪） | 将规格更新写入 `01-openspec-proposal.md` |
| 门禁 2（风险就绪） | 将 grill-me 回复写入 `02-grill-me-report.md` |
| 门禁 3（任务就绪） | 将技能映射写入 `03-task-skill-map.md` |
| 门禁 5（审查就绪） | 将审查发现写入 `05-review-report.md` |
| 门禁 6（已验证） | 将验证结果写入 `06-verification-log.md` |
| 门禁 7（知识已捕获） | 将 ADR 写入 `07-adr.md`，回顾写入 `07-task-retro.md` |

### 恢复协议关键规则

1. **优先读取制品文件，而非聊天历史。** 制品是唯一事实来源。
2. **按门禁编号递增顺序恢复。** 不要跳过未完成的门禁。
3. **若门禁制品存在但不完整，从该门禁的起点重新执行。**
4. **若当前门禁超过 7 天未活动，重新验证所有已完成的门禁。**

## 确定性门禁

为确保每次工作流执行的一致性，本工作流提供可脚本化的门禁检查机制。

- **门禁检查脚本：** `workflow/scripts/gate-check.sh` 提供了对当前功能文件夹各门禁
  制品完整性、格式正确性和内容合规性的自动化检验。编排者应在通过每个门禁后运行
  该脚本；审查者也应在审查开始前运行以确认输入质量。
- **状态模式：** `feature-state.schema.json` 定义了功能文件夹的状态模式——
  每个门禁的必需制品、它们的格式约束、以及门禁之间的依赖关系。
  该模式是门禁检查脚本的运行时依据，也可用于自定义工具链的集成。

### 门禁检查脚本用法

```bash
# 检查当前功能的全部门禁状态
bash workflow/scripts/gate-check.sh --feature <feature-id>

# 仅检查指定门禁
bash workflow/scripts/gate-check.sh --feature <feature-id> --gate 2

# 输出 JSON 格式供工具链消费
bash workflow/scripts/gate-check.sh --feature <feature-id> --json
```

脚本返回码：`0` 表示门禁通过，非零表示存在阻塞项。JSON 输出遵循
`feature-state.schema.json` 定义的 schema。

## 经验进化

工作流本身是活的——每次执行产生的经验都被系统性地捕获、评估并反馈进流程改进中。

- **经验管道：** `workflow/experience/pipeline.md` 定义了从原始经验到流程改进的
  完整管道：采集 → 聚类 → 评估 → 提案 → 合并。
  编排者在完成每个功能后（门禁 7 阶段）应将任务回顾中的经验数据提交到该管道；
  管道会定期将高置信度的改进建议合并回本工作流的核心文件。

### 管道阶段

| 阶段 | 输入 | 输出 | 负责方 |
|------|------|------|--------|
| 采集 | 任务回顾 (.md) | 原始经验条目 | 编排者 |
| 聚类 | 原始经验条目 | 按主题分组的经验簇 | 经验管道 |
| 评估 | 经验簇 | 评分 + 改进建议 | 经验管道 |
| 提案 | 改进建议 | 具体变更 PR | 经验管道 / 编排者 |
| 合并 | 变更 PR | 更新后的工作流文件 | 人类审批后合并 |

## 调度器

在 Agent 并发或多个功能并行推进的场景下，调度器负责管理工作流实例。

- **调度：** `workflow/dispatcher.md` 定义了如何在同一仓库中同时运行多个功能
  工作流而不产生冲突。它覆盖：功能 ID 命名空间隔离、制品目录隔离、
  审查和门禁的优先级排队、以及编排者 Agent 间的上下文切换协议。

### 核心调度规则

1. **功能 ID 唯一且不可变。** 每个功能文件夹以功能 ID 命名，创建后不重命名。
2. **每个功能独立一个编排者。** 一个编排者同时只能拥有一个活跃功能；
   但不同功能的编排者可以并行推进。
3. **审查队列按 P0 数量排序。** 审查者优先处理 P0 发现最多的功能。
4. **上下文切换时，编排者必须先通过恢复协议（见「循环与反馈」）重新定位。**
5. **若两个功能修改同一文件，调度器强制串行化——后创建的功能等待先创建的功能
   通过门禁 6 后方可进入门禁 3。**

## 评估与可视化

为衡量工作流执行质量并持续改进，本工作流内置了评估与可视化工具。

- **评分引擎：** `workflow/eval/scoring-engine.md` 定义了针对每次功能执行的
  量化评分体系——覆盖门禁通过率、审查发现密度、P0/P1 遗留率、时间效率、
  Agent 协作效率等维度的加权评分。评分在门禁 7 完成后自动生成。
- **仪表盘：** `dashboard.html` 提供交互式的可视化总览，展示当前仓库中所有
  功能的状态、评分趋势、审查覆盖率、瓶颈分布和团队健康度指标。
  在浏览器中直接打开即可使用，数据源为各功能文件夹中的制品文件。

### 评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 门禁合规 | 25% | 所有门禁是否完整通过 |
| 审查深度 | 20% | P0/P1 发现的密度与质量 |
| 遗留风险 | 20% | P0/P1 是否在门禁 6 前全部关闭 |
| 时间效率 | 15% | 实际耗时 vs 任务映射估算 |
| 协作效率 | 10% | Agent 间交接耗时与单 Agent 降级频率 |
| 知识产出 | 10% | ADR 与任务回顾的完整度与可操作性 |

评分引擎的详细算法和仪表盘的使用说明分别见上述两个文件。
